## The simplest way to work with Parquet in Java

Carpet maps Java records directly to Parquet files using reflection — no code generation, no schema definitions, no inheritance required. Define a record, write it, read it back.

```java QuickStart.java
record Sale(long id, String product, int quantity, double amount) {}

// Write
try (var out = new FileOutputStream("sales.parquet")) {
    try (var writer = new CarpetWriter<>(out, Sale.class)) {
        writer.write(List.of(
            new Sale(1, "Widget", 10, 299.90),
            new Sale(2, "Gadget", 5, 499.50)
        ));
    }
}

// Read
List<Sale> sales = new CarpetReader<>(new File("sales.parquet"), Sale.class).toList();
```

## Key features

<CardGroup cols={2}>
  <Card title="Zero boilerplate" icon="wand-magic-sparkles">
    No code generation, no schema files. Just Java records and Carpet handles the rest.
  </Card>
  <Card title="Nested structures" icon="sitemap">
    Full support for nested records, `List`, `Set`, `Map`, and combinations thereof.
  </Card>
  <Card title="Rich type support" icon="table">
    Covers primitives, `String`, `UUID`, `BigDecimal`, all Java time types, enums, and geospatial data.
  </Card>
  <Card title="Column projections" icon="filter">
    Read only the columns you need — Parquet skips the rest, saving I/O and CPU.
  </Card>
  <Card title="Type annotations" icon="tag">
    Control how fields are stored with `@Alias`, `@NotNull`, `@PrecisionScale`, `@ParquetJson`, and more.
  </Card>
  <Card title="Write Model API" icon="sliders">
    Map any Java class — not just records — to Parquet with custom field names and computed values.
  </Card>
  <Card title="Low-level access" icon="gear">
    Build native `ParquetWriter` / `ParquetReader` instances for third-party library integration.
  </Card>
  <Card title="Minimal dependencies" icon="feather">
    Ships without Hadoop. Only the essential `parquet-java` dependencies are included.
  </Card>
</CardGroup>
